# AI Skill 聚合展示平台 — 需求规划文档

> 版本：v1.0 | 日期：2025-07-10 | 阶段：需求规划

---

## 一、项目概述

### 1.1 产品定位

打造一个**中文友好的 AI Skill 聚合发现平台**，每日自动采集全球最新的 AI Skill（覆盖 GitHub、技能市场、技术社区等渠道），同时支持用户自主上传，通过智能分类、多维度排序和双语展示，帮助开发者和 AI 爱好者高效发现、评估和复用优质 Skill。

### 1.2 目标用户

| 用户角色 | 核心场景 | 痛点 |
|---------|---------|------|
| **AI 开发者/工程师** | 日常搜索可复用的 Skill 来增强 AI 编码助手（Claude Code、Codex、Cursor 等） | 需要在多个平台间切换搜索，信息分散，缺乏统一入口 |
| **AI 产品/爱好者** | 浏览热门 Skill 了解 AI 能力边界，发现新玩法 | 大量英文 Skill 存在语言障碍，不知从何入手 |
| **Skill 创作者** | 上传自己开发的 Skill 获取曝光和反馈 | 缺乏面向中文用户的 Skill 分发渠道 |

### 1.3 业务目标

- **内容覆盖**：上线 3 个月内收录 Skill 数量达到 10,000+
- **内容质量**：确保采集 Skill 经过基础质量筛选（非死链、有实质内容）
- **用户增长**：6 个月内 DAU 达到 1,000+

## 二、竞品分析

### 2.1 主要竞品概览

| 平台 | 收录量 | 核心特色 | 不足 |
|------|--------|---------|------|
| **SkillsMP** | 215 万+ | 按职业/创作者/分类多维度浏览，API 开放 | 全英文，国内用户门槛高 |
| **skills.sh** | — | 24 小时热度排行，由 Vercel 推出 | 仅排行，无分类体系，功能单一 |
| **Clawhub** | — | OpenClaw 官方生态，质量可靠 | 绑定特定框架，不够通用 |
| **SkillHub（腾讯云）** | — | 中文社区，本土优化 | 偏腾讯生态，覆盖面有限 |
| **虾技市场** | — | 精品路线，用户评测驱动 | 内容量少，更新频率低 |

### 2.2 差异化机会

1. **双语智能展示**：自动翻译/摘要，无论采集到英文还是中文 Skill，统一展示中文简介 + 原始名称
2. **全渠道聚合**：不只是 GitHub，覆盖技能市场、技术社区、主流媒体等全渠道
3. **中文优先**：面向中文用户优化搜索、分类和浏览体验
4. **用户贡献闭环**：采集 + 上传双轮驱动，用户可以补充平台未覆盖的 Skill
5. **质量筛选机制**：不只堆数量，通过星标数、更新频率、社区反馈等维度做质量加权

## 三、功能清单

### 3.1 核心功能（MVP）

#### F1. Skill 首页卡片展示
- 以卡片网格形式展示 Skill 列表
- 每张卡片展示：Skill 名称、中文简介、分类标签、来源渠道、热度/时间、星标数
- 支持瀑布流或网格布局

#### F2. 多维度排序
- **最新排序**：按采集/发布时间倒序
- **热度排序**：综合热度算法（星标数 + 浏览量 + 收藏量 + 时间衰减因子）
- 默认展示热度排序

#### F3. Skill 双语展示规则
- Skill 名称保留原始语言（英文保持英文，中文保持中文）
- 简介统一展示中文版本（英文 Skill 自动翻译摘要）
- 详情页提供原始描述 + 中文翻译对照

#### F4. Skill 分类体系
- 一级分类（建议）：开发工具、数据处理、内容创作、设计创意、办公效率、安全测试、商业运营、其他
- 支持按分类筛选和浏览
- 每个 Skill 支持多标签

#### F5. Skill 详情页
- 完整描述（原始 + 中文）
- 来源链接（GitHub 等）
- 基础元数据：作者、更新时间、适用平台、安装方式
- 相关 Skill 推荐

#### F6. 自动采集系统
- 定时任务每日自动抓取
- 采集渠道：GitHub 公开仓库、技能市场 API、技术社区 RSS
- 去重机制（基于 URL 或 Skill 唯一标识）
- 新入库 Skill 自动标记"NEW"

#### F7. 用户上传功能
- 提交表单：名称、描述、分类、来源链接、适用平台
- 上传后进入待审核状态
- 管理员审核通过后公开显示

|------|------|--------|

| API 开放 | 提供 REST API 供第三方调用 | P3 |

## 四、核心流程

### 4.1 用户浏览流程

```
用户访问首页
  → 默认按热度排序展示 Skill 卡片
  → 可切换「最新」/「最热」排序
  → 可点击分类标签筛选
  → 点击卡片进入详情页
  → 查看完整信息 + 来源链接
  → （登录后）可收藏/点赞
```

### 4.2 自动采集流程

```
定时任务触发（每日定时）
  → 遍历采集源列表
  → 抓取新 Skill 数据
  → 去重检查
  → 英文 Skill → 自动翻译摘要
  → 分类自动匹配（基于关键词/AI分类）
  → 写入数据库
  → 标记"NEW"标签（24小时内）
```

### 4.3 用户上传流程

```
用户点击「上传 Skill」
  → 填写表单（名称、描述、分类、链接等）
  → 提交 → 状态：待审核
  → 管理员后台审核
  → 通过 → 公开显示 / 驳回 → 通知用户修改
```

## 五、数据指标

### 5.1 核心指标

| 指标 | 定义 | 目标（3个月） |
|------|------|-------------|
| Skill 总量 | 平台上可浏览的 Skill 总数 | ≥ 10,000 |
| 日新增 Skill | 每日新入库的 Skill 数量 | ≥ 30 |
| 日均 UV | 每日独立访客 | ≥ 500 |
| 卡片点击率 | 首页卡片点击 / 首页曝光 | ≥ 15% |
| 来源链接点击率 | 点击跳转原始来源的比例 | ≥ 25% |

### 5.2 质量指标

| 指标 | 定义 |
|------|------|
| 采集成功率 | 每日成功采集数 / 计划采集数 |
| 去重准确率 | 正确识别重复 Skill 的比例 |
| 翻译可用率 | 机器翻译后人工可读的比例 |
| 死链率 | 详情页来源链接 404 的比例 |

### 5.3 埋点需求（建议）

- 首页曝光：卡片曝光、排序切换、分类筛选点击
- 详情页：页面 PV、来源链接点击、停留时长
- 上传：提交成功率、审核通过率
- 搜索：搜索关键词、搜索结果点击

## 六、技术方案概要

### 6.1 推荐技术栈

| 层 | 技术选型 | 理由 |
|----|---------|------|
| 前端 | Next.js + Tailwind CSS | SEO 友好、生态成熟、部署方便 |
| 后端 | Node.js / Python FastAPI | 看团队技术栈偏好 |
| 数据库 | PostgreSQL + Redis | 结构化数据 + 热度缓存 |
| 搜索 | Elasticsearch / Meilisearch | 中文分词搜索 |
| 采集 | Python 定时任务 + GitHub API | 爬虫生态丰富 |
| 翻译 | OpenAI / DeepL API | 自动翻译摘要 |
| 部署 | Vercel / 阿里云 | 前端 Vercel 免费额度友好 |

### 6.2 热度算法建议

```
热度 = (星标数 × 0.4 + 浏览量 × 0.2 + 收藏量 × 0.3 + 评论数 × 0.1) × 时间衰减因子
时间衰减因子 = 1 / (1 + α × 天数)  # α 建议取值 0.05-0.1
```

## 七、待确认问题

在进入 PRD 撰写前，建议先明确以下问题：

| # | 问题 | 建议 |
|---|------|------|
| 1 | 技术栈偏好？前端/后端用什么框架？ | 建议 Next.js + Python，但需确认团队能力 |
| 2 | 是否需要用户系统？MVP 阶段就上还是后续？ | 建议 MVP 先做浏览 + 上传（无需登录也可上传），V1.1 再加用户系统 |
| 3 | 翻译是机器翻译还是人工？ | 建议机器翻译 + "人工优化"入口 |
| 4 | 采集频率？每日几次？ | 建议每日 1-2 次，避免 API 限流 |
| 5 | 是否需要后台管理系统？ | MVP 至少需要一个简易审核后台 |
| 6 | 域名和部署环境是否已有？ | 需确认 |
| 7 | 团队有多少人？前后端分工？ | 影响 MVP 范围和时间规划 |
